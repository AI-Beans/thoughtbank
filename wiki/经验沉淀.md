## 一、Ingest 经验

### 来源标注规范

```markdown
来源: raw/目录名/文件名.扩展名
```

### 默会知识外化

发现无法被规则捕获的判断瞬间时：
- 记录下来
- 存入 outputs/
- 最终沉淀到本页面

### 常见错误与修正

| 错误 | 发现方法 | 修正 |
|---|---|---|
| 把公司项目当独立项目 | 读源码 | 从代码注释提取公司名 |
| 时间线混乱 | 追问确认 | 明确并存 vs 先后 |
| 主题边界不清 | 自问"能一句话说清吗" | 拆分或合并 |

### 深度 Lint 实践（raw/ 漏读防范）

| 错误 | 发现方法 | 修正 |
|---|---|---|
| node_modules 污染统计 | 发现 8516 文件中数千个是依赖 | 排除 node_modules、.dll、.exe、.cache |
| 深层目录漏读 | 对比目录数 vs 实际读取数 | 对 Level 2+ 做针对性深度 ingest |
| 散文件未分类 | Level 1 有 60 个无目录文件 | 按日期/类型分组或关联到已有页面 |

## 二、工具使用策略

### 降维打击策略（大型工具用于小型项目）

**经验法则**：用企业级工具做独立项目可以降维打击，但必须取精华不全搬。

| 取 | 不取 | 原因 |
|---|---|---|
| 项目宪法 | 完整六阶段流水线 | 宪法一次写好永久生效，流水线太重 |
| Phase Gates | Library-First 原则 | 关键变更前自检有价值，强制库化不适合原型 |
| 规范持久化 | CLI Interface Mandate | 需求可追溯有价值，强制 CLI 接口过度约束 |

### 工具桥接方法论

当两个工具格式兼容但无原生集成时：
1. 确认格式兼容性（都是 Markdown + checkbox）
2. 写桥接脚本做格式转换
3. 在 AGENTS.md 中声明集成流程
4. 实践验证后开源贡献

## 三、设计原则

### AGENTS.md 纯净度原则

**发现场景**: 在 ThoughtBank 维护过程中发现 AGENTS.md 混合了两种性质不同的内容——"操作规范"和"实践真知"。

**原则**: AGENTS.md 应只保留 Schema 规范（"做什么、怎么做"），实践真知（"踩过什么坑、发现什么规律"）放到独立的 wiki 页面。Agent 每次启动都读 AGENTS.md，它应该是操作手册，不是博客。

### 播种不包收原则

**发现场景**: harness-init skill 实战验证——83 行 AGENTS.md 驱动 OmC 生成了 281 行完整实施计划。

**原则**: 初始化工具的价值在于播种，之后的生长是 Agent 框架的能力。好的模板给框架 + 留白，不需要穷举所有可能。
