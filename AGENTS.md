# 知识库 Schema

## 这是什么

[你的主题] 的个人知识库，由 LLM 维护。

**核心理念**: 先把知识编译成持续存在的中间层，而不是每次提问时临时检索。

## 目录结构

```
./
├── AGENTS.md      # 本文件（OpenCode 读取的维护规范）
├── index.html     # 知识图谱可视化（浏览器打开）
├── raw/           # 原始材料（不可变）→ 源文档收纳
├── wiki/          # AI 整理的维基 → 由 AI 维护的笔记
│   ├── index.md  # 内容目录
│   └── log.md    # 时间线日志
└── outputs/       # AI 生成物 → 答案/报告/分析
```

## 核心理念

> **默会知识**：我们知道的多于能表达的。知识要积累，不要每次重新发现。

## 工作流程（完整闭环）

```
┌─────────────────────────────────────────┐
│  RAW — 源素材放入                         │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  INGEST — 分析 → wiki → index → log    │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  QUERY — 提问 → wiki → 回答             │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  FILE BACK — 有价值的回答 → outputs/    │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  LINT — 健康检查 → 修正 → 记录         │
└─────────────────────────────────────────┘
```

## 自动抓取网页（agent-browser）

**agent-browser**（Vercel Labs）— AI 浏览器操控工具，让 AI 直接抓取任意网页内容到 `raw/`。

```bash
# 安装
npm install -g agent-browser
npx agent-browser install

# 使用示例
你: "把这个 URL 抓下来存到 raw/：https://example.com/article"
AI:  agent-browser open https://example.com/article
     agent-browser get text "article"
     # AI 打开页面，抓取文本，保存到 raw/
```

**优势**：
- 告别手动复制粘贴
- 处理 JS 动态加载、需登录、交互式图表的页面
- 比 Playwright MCP 省 **82% token**

**适用场景**：竞品文章、热门话题、研究文档、社交媒体讨论。

## Ingest 规范

### 执行步骤

1. 将源文件放入 `raw/`（手动放入 / agent-browser 抓取 / 复制粘贴）
2. LLM 分析内容（**要读源码，不只是文件名**）
3. 识别主题，归纳到已有页面或新建页面
4. 更新相关页面的链接关系
5. 更新 `wiki/index.md`
6. 追加到 `wiki/log.md`

### 深度 ingest 检查清单

- [ ] 读源码逻辑（不只是 README）
- [ ] 从代码注释中提取真实背景
- [ ] 识别公司/产品/项目名称
- [ ] 修正浅层假设
- [ ] 确认时间线关系（并存 vs 先后）

### 关键原则

> **不要假设，要验证。**
>
> 浅层 ingest 会把"公司产品"当成"学习项目"，把"公司名"当成"个人品牌"。

## Query 规范

### 执行步骤

1. 用户提问
2. LLM 读取 `index.md` 定位相关页面（**不要直接读 raw**）
3. LLM 读取页面内容，综合回答
4. 有价值的分析/报告存到 `outputs/`
5. 如果回答揭示了新关联，更新相关 wiki 页面

### 决策标准：什么值得回答

| 类型 | 处理 |
|---|---|
| yes/no 问题 | 直接回答，不存 |
| 单一页面能回答 | 直接回答，可能存 |
| **跨多页综合** | **必须存 outputs/** |
| 揭示新关联 | **必须更新 wiki** |

## File Back 规范

**每次 Query 产出两份**：
- 一份回答用户（直接返回）
- 一份回写到 `outputs/` 或更新 wiki

## Lint 健康检查

**定期执行**，检查：
1. 页面间矛盾
2. 提到但未解释的主题（幽灵主题）
3. 无源支持的声明
4. 孤立页面（无 inbound 链接）
5. 建议填补空白的新来源

## Wiki 页面规范

### 页面结构

```markdown
# 主题名称

一句话摘要（核心定义，1-2句）

## 内容...

## 来源
- `raw/xxx/` — 说明

## 关联主题
[[主题1]]
[[主题2]]
```

### 链接原则

- 每个页面 2-5 个相关主题链接
- 使用 `[[topic-name]]` 双向链接
- **链接要有意义**，不是形式链接

### 内容规范

- 每个声明尽量注明 `raw/` 中的具体文件
- 提到但未解释的概念 → 创建独立页面或添加一句话定义
- **不手工编辑维基**，那是 AI 的工作

## index.md 维护

按类别组织，每项包含链接 + 一句话描述：

```markdown
## Entities / 实体
- [[主题名]] — 一句话描述

## Concepts / 概念
- [[概念名]] — 一句话描述
```

## log.md 规范

Append-only，每次 ingest/query/lint 都记录：

```markdown
## [YYYY-MM-DD] 操作类型 | 简短描述

**来源**: ...
**处理内容**: ...
**新增页面**: ...
**更新页面**: ...
```

## 兴趣点

[列出 3-5 个你希望这个知识库关注的方向]

示例：
1. **领域A** — 描述
2. **领域B** — 描述
3. **领域C** — 描述

## 输出格式

- Markdown 页面（存入 wiki/）
- 对比表格
- 简报/报告（存入 outputs/，命名：`YYYY-MM-DD-主题分析.md`）
- Marp 幻灯片

## 经验沉淀

### 来源标注规范

```markdown
来源: raw/目录名/文件名.扩展名
```

### 默会知识外化

发现无法被规则捕获的判断瞬间时：
- 记录下来
- 存入 outputs/
- 最终沉淀到方法论

### 常见错误与修正

| 错误 | 发现方法 | 修正 |
|---|---|---|
| 把公司项目当独立项目 | 读源码 | 从代码注释提取公司名 |
| 时间线混乱 | 追问确认 | 明确并存 vs 先后 |
| 主题边界不清 | 自问"能一句话说清吗" | 拆分或合并 |

### 深度 Lint 实践（raw/ 漏读防范）

| 错误 | 发现方法 | 修正 |
|---|---|---|
| node_modules 污染统计 | 发现 8516 文件中数千个是依赖 | 排除 node_modules、.dll、.exe、.cache |
| 深层目录漏读 | 对比目录数 vs 实际读取数 | 对 Level 2+ 做针对性深度 ingest |
| 散文件未分类 | Level 1 有 60 个无目录文件 | 按日期/类型分组或关联到已有页面 |

### 降维打击策略（大型工具用于小型项目）

**经验法则**：用企业级工具做独立项目可以降维打击，但必须取精华不全搬。

| 取 | 不取 | 原因 |
|---|---|---|
| 项目宪法 | 完整六阶段流水线 | 宪法一次写好永久生效，流水线太重 |
| Phase Gates | Library-First 原则 | 关键变更前自检有价值，强制库化不适合原型 |
| 规范持久化 | CLI Interface Mandate | 需求可追溯有价值，强制 CLI 接口过度约束 |

### 工具桥接方法论

当两个工具格式兼容但无原生集成时：
1. 确认格式兼容性（都是 Markdown + checkbox）
2. 写桥接脚本做格式转换
3. 在 AGENTS.md 中声明集成流程
4. 实践验证后开源贡献
